# Deepen the host upload endpoint behind one function

Status: accepted (2026-06-18)
Scope: `web/src` — new `upload-endpoint/`, `server.ts`, `worker.ts`, `rate-limiter.ts`, `providers/`, `tests/`

After ADR-0001 collapsed the provider proxy runtime and ADR-0002 extracted the client sequencers, the three hosts (`server.ts`, `worker.ts`, `rate-limiter.ts`) still each carried their own copy of the *upload endpoint* behaviour: route matching, form-data reading, token resolution, validation gating, the provider/engine call, status normalization, the JSON error envelope, CORS application, and rate-limit header projection. These had already drifted — the Node host skipped catbox/imgchest/sxcu-files validation that the Worker/DO applied; the rate-limit header projection existed twice under two names (`buildRateLimitHeaders`, `createResponseHeadersFromProvider`); the DO's imgchest token resolution used a fragile `replace('Bearer ', '')` and leaked a debug blob with a truncated token. We are collapsing all of that behind one deep module.

## Decision

- **One function, one interface.** A new module at `src/upload-endpoint/` exports `handleUploadRequest(request, deps): Promise<Response>` and nothing else. It owns route matching, request shaping (form-data reading, token resolution, validation), the provider/engine call, and response shaping. Each host reduces to a single call to it, plus its own runtime wiring. This is the deepest possible module — smallest interface, largest implementation — and matches the architecture review's "After" diagram.
- **`deps` carries only resolved host-runtime facts, never logic.** `{ corsHeaders, fetch?, store?, secrets }`. `corsHeaders` is pre-resolved by the host (CORS origin allow-listing is a host security policy, not upload logic). `secrets` arrive pre-resolved (`getBearerToken(req) ?? env token`); the module never reads `process.env` or `request.env`. `store` is optional — catbox/kek ignore it, imgchest/sxcu require it (the module errors internally if missing, which is unreachable in practice). No route table, no per-provider callbacks, no `env` passthrough: the route catalog stays inside.
- **Request shaping moves out of the hosts and out of `providers/`.** `request-shaping.ts` owns one reader per provider (`readCatboxRequest`, `readKekRequest`, `readSxcuRequest`, `readImgchestRequest`). The existing `readCatboxUploadInput` / `readKekUploadInput` relocate from `providers/` to here; new readers for sxcu and imgchest replace the Node/DO inlined copies. Validation is baked into each reader, which is what makes validation uniform across all hosts. Readers return a tagged `{ ok, input } | { ok: false, error }` union — no exceptions across the seam. This realigns `providers/` with ADR-0001's stated contract (pure "given an input + injectable fetch, return a `ProviderResult`"); the readers were a placement accident, not a designed boundary.
- **Response shaping consolidates.** `response-shaping.ts` owns per-provider status normalization (2xx→200 for catbox/kek; passthrough for imgchest/sxcu — a provider rule, not host drift), a single `errorResponse(status, message, corsHeaders)` helper replacing ~15 hand-built `{ error }` copies, and a single rate-limit header projection replacing both `buildRateLimitHeaders` and `createResponseHeadersFromProvider`.

## The non-obvious part: the module is placement-agnostic, but the Worker still forwards to the DO

A natural reading of "one deep module called by all hosts" would push the Worker→Durable-Object forwarding *into* the module (e.g. a `rateLimiterNamespace?` field on `deps`). **We deliberately did not do that.** The DO forwarding is not upload behaviour — it is ADR-0001's per-IP atomicity mechanism (`idFromName('client-<IP>')`, `X-Origin`/token injection, `rateLimiter.fetch`). Pulling it into the module would force a topology branch ("if imgchest/sxcu and namespace provided, forward; else call providers") into the upload-logic module, leak the `RATE_LIMITER` binding across the seam, and make the module untestable without a DO-namespace mock.

Instead the **same** `handleUploadRequest` is the entry point in two places: the Worker calls it for catbox/kek, and the Durable Object calls it for imgchest/sxcu (with `store = new DurableObjectRateLimitStore(this.storage)`). The Worker→DO split stays in the Worker, where ADR-0001 put it. The module never sees the DO binding. ADR-0001 is untouched.

## Preservation requirements (behaviour that must not flatten during extraction)

Mirroring how ADR-0002 called out the three imgchest forks, these existing behaviours are preserved *exactly* and must not be "tidied":

- **Catbox raw-text success body.** catbox returns `String(result.body)` with no content-type — the client expects a bare URL string, not JSON. This stays a per-provider shaping rule, not a default.
- **Strictest validation wins (closes Node's gaps).** catbox via `validateFiles`, imgchest via `validateImgchestFiles`, sxcu-files via `validateFiles`, applied uniformly. Node previously skipped all three; that was an oversight, not policy. The proxy's own caps (catbox `MAX_FILE_SIZE=50MB`, imgchest `IMGCHEST_MAX_FILE_SIZE=30MB`) stay stricter than the upstream docs (catbox; sxcu 95MB) where they already are.
- **sxcu 429 global-detection side-effect** (`providers/sxcu.ts`): when the body carries `code === 2`, `rateLimitHeaders.isGlobal` is rewritten. This happens inside the provider before response shaping; the projection merely reads it.
- **imgchest defaults and post-PATCH step.** `nsfw` defaults `true`, `privacy` defaults `hidden`. After add-to-post with privacy/nsfw set, `uploadToImgchest` issues a follow-up PATCH on the post. Both are existing behaviour; keep.
- **sxcu collections are not engine-gated.** The sxcu.net docs mark collection-create as `2 req./min.`, but our engine does not gate it today (`engine: null`). Preserved as-is; gating it is a separate follow-up, not part of this refactor.
- **DO imgchest token debug blob deletes.** `rate-limiter.ts`'s failure response with `debug: { hasAuthHeader, authHeaderValue: token.substring(0,20)+'...' }` is removed — it leaked a truncated token. Canonical resolution is `getBearerToken(req) ?? secrets.imgchestToken` (header wins), no `replace('Bearer ', '')`.

## Considered options (rejected)

- **Router + exported per-route handlers.** Rejected: a wider interface (every handler exported) makes a shallower module — the host still "knows" the route list. The whole point is one function.
- **Place the module under `providers/` (e.g. `providers/router.ts`).** Rejected: `providers/` means "talks to an upstream provider API and returns a `ProviderResult`" (per `CONTEXT.md`); the new module talks to `Request`/`Response` and orchestrates providers. Co-locating would blur the layer boundary exactly as ADR-0002 warned against on the client side.
- **Place the module under `src/server/` or name it "server"/"controller".** Rejected: clashes with the existing root `server.ts` (the Node host) and "server" is a forbidden glossary synonym for host.
- **Absorb the Worker→DO forwarding into the module via a `rateLimiterNamespace?` dep.** Rejected for the atomicity/topology reason above.
- **Always route through the rate-limit engine with a no-op policy/store for catbox/kek.** Rejected: catbox/kek genuinely do not rate-limit; a no-op store invents a concept ("passthrough store") only to make the code uniform, hiding a real semantic difference.
- **Leave catbox/kek input readers in `providers/`, only move sxcu/imgchest.** Rejected: splits "where does formData→input live?" across two folders by provider; a future reader adding a provider won't know which folder. One module, one home.
- **Async-generator / event seam for the endpoint.** Rejected: out of scope — this is a synchronous request→response transform, not a streaming flow (ADR-0002 already settled the generator question for the client side).

## Consequences

- `server.ts` `handleRequest` keeps static-file serving, the 404 fallback, and the Node→web `Request`/`Response` bridging; the upload handlers delete and become one `handleUploadRequest` call.
- `worker.ts` keeps OPTIONS preflight, the 413 request-size gate, and the DO-forwarding branch for imgchest/sxcu; its own `handleCatboxUpload` / `handleKekUpload` delete and delegate to the module. **The pre-flight `if (!env.RATE_LIMITER)` guard stays** — it sits on the DO-forwarding path itself, which the Worker keeps (catbox/kek are handled by the module before the guard; imgchest/sxcu fall through to the guard and then forward). It still fires when someone deploys without the `RATE_LIMITER` binding, so it is not dead code.
- `rate-limiter.ts` `RateLimiter.fetch` keeps the per-IP placement (decided in the Worker) and the engine-in-DO atomicity (ADR-0001); its four private handlers collapse into one `handleUploadRequest` call with a `DurableObjectRateLimitStore`.
- `providers/` shrink: `readCatboxUploadInput` / `readKekUploadInput` move out; providers become purely input→fetch→`ProviderResult`.
- A new `tests/upload-endpoint.test.ts` becomes the primary surface — one suite asserting every shaping rule once (route matching, validation, status normalization, error envelope, CORS application, rate-limit header projection, engine-or-not branch), using an injected `fetch` double and an in-memory store. `tests/server.test.ts`, `tests/worker.test.ts`, and `tests/rate-limiter.test.ts` shrink to genuinely host-scoped concerns (static files; OPTIONS/413/DO-forward; per-IP sharding and `DurableObjectRateLimitStore`). The same upload logic is no longer tested three times under three shells.
- Follow-up (open, out of scope here): gate sxcu collection creation behind the engine to match the upstream `2 req./min.`; and candidate-2 from the architecture review remains — exercising the real Worker/DO runtime seam via `@cloudflare/vitest-pool-workers` (still deferred per ADR-0001).
