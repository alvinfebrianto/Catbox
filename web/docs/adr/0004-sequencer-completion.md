# Make upload sequencer completion part of the interface

Status: accepted (2026-06-18)
Scope: `web/src` — `upload/contracts.ts`, `upload/catbox.ts`, `upload/kek.ts`, `upload/imgchest.ts`, `upload/sxcu.ts`, `app.ts`; `web/tests` — `upload/*.test.ts`, `upload/recording-observer.ts`, `app.test.ts`

After ADR-0002 extracted the client-side upload sequencers behind an observer seam, the sequencers were deep for upload *flow* but shallow about *lifecycle*: each returned `void` and signalled completion only through `observer.onDone(results)`. That made completion a fire-and-forget callback the caller could not await, with two visible consequences. First, a bug: `ImageUploader.handleSubmit` wraps the dispatch in `try { ... } finally { setLoading(false); progressDiv.style.display='none' }`, but because the dispatch returns synchronously, the `finally` runs *immediately* — the spinner disappears and the submit button re-enables while uploads are still in flight, and the only real completion signal (`onDone` → `displayResults`) never restores loading state. Second, the tests could not assert completion directly: they poll `observer.doneWith` via `vi.waitFor` instead. We are making completion part of the interface.

This is architecture review candidate-3 ("Make upload sequencer completion part of the interface"), the next Strong, in-process candidate after candidate-1 (ADR-0003) landed.

## Decision

- **Sequencers return `Promise<UploadResult[]>`.** The interface becomes `uploadToCatbox(input, observer, fetchFn): Promise<UploadResult[]>` (and likewise for kek, sxcu, imgchest). Resolution = completion, with the full results array. The promise is the single completion signal. This is the idiomatic reading of the review's "events + completion + failure" and directly fixes the finally-runs-immediately bug: the caller `await`s the sequencer, so cleanup in `finally` runs at completion.
- **The observer retires `onDone` and shrinks to incremental events only.** `UploadObserver` becomes `{ onResult, onProgress, onRateLimitWait }`. `onDone(results)` carried the exact payload the promise now resolves with — a redundant second completion signal that invited drift. Removing it makes the observer purely incremental and the promise purely terminal.
- **Per-item errors stay data in the resolved array; rejection is for the unexpected only.** A single file failing, imgchest's first-file-terminates-the-post, catbox's album-creation failure, and imgchest's validation failure are all flows that *complete* today (they reach `onDone` with the error folded into the results array as `{ type: 'error', message }`). They continue to **resolve** with the same array — a partial upload is a completed flow, not a failed one. The promise **rejects** only for an exception the sequencer did not convert to per-item data (a bug); today those are swallowed, after this change the host's `catch` surfaces them as a real `showError`.
- **`handleSubmit` and the four provider wrappers become `async`.** Async propagates end-to-end so the `await` lands on real completion. `handleSubmit` does `const results = await this.uploadToCatbox(urls); this.displayResults(results, totalItems);` inside the existing `try`, and the single `finally` (`setLoading(false)` + hide progress) now runs at completion. `displayResults` moves from the `onDone` bridge (`makeObserver`) to the await site. `makeObserver` returns an observer with only the three incremental methods.
- **Minimal internal change: wrap, do not rewrite.** Each sequencer's internal control flow stays verbatim (catbox/kek/imgchest callback chains, sxcu's existing async/await). The entry is wrapped in `new Promise<UploadResult[]>((resolve, reject) => { ... })`; every `observer.onDone(results)` call becomes `resolve(results)`; the existing per-item `.catch`/`try` handlers keep building the error result and now `resolve(results)` too. ADR-0002's preserved forks (catbox continues on per-item error; imgchest first-file terminates; sxcu burst pacing) stay byte-for-byte.

## The non-obvious parts

**The `uploadToImgchest` early-exit guard hoists into `handleSubmit`.** The wrapper's `if (postId && anonymous)` guard (`app.ts`) today calls `showError(...)` plus manual `setLoading(false)`/hide-progress and returns — a second copy of the cleanup that exists only because the shared `finally` runs too early for a synchronous early-return. Under the new chain the wrapper would return `Promise.resolve([])` and `handleSubmit` would then call `displayResults([], totalItems)`, which renders a "Successfully uploaded 0 out of 0 files." summary *over* the `showError` message — a double-output bug. The fix is structural: hoist the guard into `handleSubmit` *before* `setLoading(true)`, so the early-exit returns with `showError` and never touches loading or `displayResults`. The wrapper's guard and its manual cleanup delete. This is a host-level input check (like the Worker's 413 size gate that ADR-0003 kept as a host concern), not sequencer logic.

**An unexpected rejection cannot escape the event listener.** `handleSubmit` is registered as `this.form.addEventListener('submit', (e) => this.handleSubmit(e))` — the arrow discards the returned promise. The internal `try/await/catch` is the real boundary (it catches the awaited rejection and calls `showError`), but to guarantee no unhandled rejection escapes if the catch/finally itself throws, the registration arrow becomes `(e) => { void this.handleSubmit(e).catch(err => console.error('upload handler failed', err)); }`.

**Rejection is not fail-fast.** It would be tempting to reject whenever *any* item errored (reject if `results` contains a `{ type: 'error' }`). That changes observable behaviour: a flow that "completes" today (1 of 3 uploads fails, the other 2 succeed) would reject, and `handleSubmit` would call `showError` instead of `displayResults`, hiding the 2 successful results. Rejected — a regression dressed up as fail-fast.

## Preservation requirements (behaviour that must not flatten during extraction)

Mirroring ADR-0003's call-outs, these existing behaviours are preserved *exactly* and must not be "tidied" into the completion shape:

- **Per-item errors are data.** Every path that today reaches `onDone` with error results folded into the array (catbox per-file failure → continue; kek per-file failure → continue; imgchest first-file-terminates-the-post; imgchest validation failure; catbox album-creation failure) **resolves** with the same array. None of them reject.
- **The observer's incremental events are unchanged.** `onResult`, `onProgress`, `onRateLimitWait` keep their exact signatures, firing order, and payloads. Only `onDone` is removed.
- **`fetchFn` injection is unchanged.** Tests keep stubbing the upstream via the third argument exactly as today.
- **`displayResults` and the other DOM methods of `ImageUploader` are not touched.** Rendering stays where ADR-0002 put it (DOM ownership in the class); only the *call site* of `displayResults` moves from `onDone` to the await resolution. Result-rendering deepening (review candidate-5) is explicitly out of scope.
- **The `uploadToImgchest` `postId && anonymous` error message is preserved verbatim.** Only its location moves.

## Considered options (rejected)

- **`Promise<void>` with `onDone` retained.** Rejected: two completion signals for one completion (the resolved promise *and* `onDone`), and the caller must wire both. The same redundancy that made `Promise<UploadResult[]>` win over `Promise<void>` argues against keeping `onDone`.
- **Reject when any item errored.** Rejected: changes observable behaviour, hides successful results on a partial-failure flow (see above).
- **Never reject; swallow all exceptions internally.** Rejected: a genuine sequencer bug would resolve with partial/empty results and no error surfaced — strictly worse than today for diagnosability. Rejection-for-the-unexpected is what makes the `handleSubmit` `catch` stop being dead code.
- **A lifecycle handle object (`{ done, cancel? }`).** Rejected: heavier interface; cancellation is speculative (no caller needs it today). Violates "smallest interface" unless cancellation is a real requirement.
- **Rewrite catbox/kek/imgchest internals to async/await while in there.** Rejected: out of proportion to "make completion part of the interface", and a behavioural rewrite re-opens verification of the ADR-0002 preserved forks under a rewrite. The minimal wrap leaves the forks byte-for-byte.
- **Reject from the `uploadToImgchest` wrapper on `postId && anonymous`.** Rejected: overloads the rejection channel. We settled that rejection means "unexpected exception (a bug)"; a user misconfiguring `postId` + `anonymous` is a normal, fixable input error, not a bug. Conflating the two muddies the rule. The guard hoists into `handleSubmit` instead.
- **Patch `displayResults` to no-op on empty + error-shown.** Rejected: `displayResults` should not know about a sibling `showError` call; it weakens the method's contract for one caller's edge case.

## Consequences

- `upload/contracts.ts`: sequencer signatures change to `Promise<UploadResult[]>`; `UploadObserver` loses `onDone`.
- `upload/catbox.ts`, `upload/kek.ts`, `upload/imgchest.ts`, `upload/sxcu.ts`: entries wrap in `new Promise`; the terminal `observer.onDone(results)` calls (15 in total across the four files) become `resolve(results)`; the per-item error handlers resolve with the error folded into the array. `observer.onDone` is no longer referenced.
- `app.ts`: `handleSubmit` and the four provider wrappers become `async`; `displayResults` moves to the await site; `makeObserver` returns the three-method observer; the `uploadToImgchest` `postId && anonymous` guard hoists into `handleSubmit` before `setLoading(true)` and its wrapper-side guard + manual cleanup delete; the submit-listener registration gains a defensive `.catch`.
- `tests/upload/recording-observer.ts`: `doneWith`, `onDone`, and the `'done'` event variant delete.
- `tests/upload/*.test.ts`: every `uploadToCatbox(input, observer, fetch); await vi.waitFor(() => expect(observer.doneWith).toHaveLength(1));` becomes `const results = await uploadToCatbox(input, observer, fetch);`, asserting on the returned array instead of `observer.doneWith[0]`. The sxcu fake-timer tests keep their `vi.advanceTimersByTimeAsync` + `flushPromises` to drive the async internals, then `await` the promise.
- `tests/app.test.ts`: the wrapper stubs return `Promise.resolve([])`; the synchronous `handleSubmit` calls become `await uploader.handleSubmit(...)`; a new test asserts the bug fix — the button stays disabled and the progress bar stays visible while the sequencer promise is pending, then both clear on resolution (an assertion that is *impossible* against today's code and would currently fail).
- Follow-ups (open, out of scope here): review candidate-2 (real Worker/DO runtime test adapter via `@cloudflare/vitest-pool-workers`, still deferred per ADR-0001); candidate-4 (provider capability rules); candidate-5 (result rendering). None are unblocked or blocked by this change.
